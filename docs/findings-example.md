# Пример отчёта по качеству данных

Ниже — какие нарушения ловят проверки на «грязном» наборе `db/seed-anomalies.sql`.
Так выглядит результат прогона `scripts/run_checks.sh` на проблемной выгрузке CRM.

| Проверка | Нарушение в данных | Что значит для бизнеса |
|----------|--------------------|------------------------|
| `check_01_duplicate_customer_emails` | клиенты 1 и 2 с одним email (`ivan@example.com`) | дубли клиентов → задвоенная статистика и рассылки |
| `check_02_orphan_bookings` | заявка 4 ссылается на клиента 99 (нет в БД) | «висящая» заявка без владельца |
| `check_03_orphan_payments` | платёж 3 привязан к заявке 777 (нет в БД) | потерянная привязка оплаты |
| `check_04_paid_without_succeeded_payment` | заявка 2 в статусе `paid`, но платёж `failed` | статус не подтверждён деньгами |
| `check_05_payment_amount_mismatch` | заявка 1: сумма 170000, оплачено 90000 | недоплата / расхождение в учёте |
| `check_06_non_positive_amounts` | заявка 6 с суммой 0 | битая/тестовая запись в проде |
| `check_07_pax_sanity` | заявка 3: 5 туристов при вместимости 2 | перепродажа мест |
| `check_08_invalid_status` | заявка 3 со статусом `confirmed` (нет в словаре) | рассинхрон статусной модели |
| `check_09_refunded_without_refund_payment` | заявка 5 `refunded` без платежа `refunded` | возврат не проведён по деньгам |
| `check_10_future_timestamps` | заявка 6 с датой `2099-01-01` | битое поле даты |

Такие SQL-выборки прикладывались к баг-репортам как доказательство и помогали
быстрее локализовать корень проблемы в данных.
